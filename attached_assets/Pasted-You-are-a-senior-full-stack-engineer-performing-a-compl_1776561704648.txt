TASK 1 — OFFLINE MODE (HIGHEST PRIORITY)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Offline mode is the most important feature in this build. The app must work fully even when the internet drops, the router restarts, or the device loses connectivity mid-session. This is non-negotiable for a café environment.

## 1A — Architecture

Use IndexedDB via Dexie.js (already approved) as the local database. Every write to Supabase must go through a smart write helper that decides the path based on connectivity.

File structure to create:
  /lib/offline/db.ts         — Dexie schema: sessions, consumptions, products, clients, outbox
  /lib/offline/queue.ts      — Outbox: enqueue, dequeue, retry logic
  /lib/offline/sync.ts       — Sync runner: processes outbox when connectivity returns
  /lib/offline/writeHelper.ts — Smart write: online → Supabase directly; offline → IndexedDB + outbox
  /shared/hooks/useConnectivity.ts — { isOnline, lastSyncAt, syncStatus }

## 1B — IndexedDB Schema (Dexie)

Define these stores:
  sessions:             id, client_id, venue_id, billing_mode, start_time, end_time, status, final_amount, rate, rate_unit, synced, created_at
  session_consumptions: id, session_id, product_id, name, quantity, unit_price, synced, created_at
  products:             id, venue_id, name, price, category, cached_at
  clients:              id, venue_id, name, phone, cached_at
  outbox:               id (autoincrement), action, payload (JSON), status ('pending'|'failed'), retries, created_at, last_attempt

## 1C — Connectivity Detection

Do NOT rely solely on navigator.onLine — it lies. Use a two-layer check:
  Layer 1: listen to window 'online' and 'offline' events
  Layer 2: every 30 seconds, ping Supabase with a lightweight query (SELECT 1 FROM spaces LIMIT 1)
  If the ping fails but navigator.onLine is true → treat as offline
  If both agree online → confirm online

Expose via useConnectivity():
  { isOnline: boolean, lastSyncAt: Date | null, syncStatus: 'idle' | 'syncing' | 'error' }

Show a persistent banner at the top of every page when offline:
  "Hors ligne — Les données seront synchronisées dès le retour de la connexion"
  Include a spinner when syncing and a checkmark when last sync succeeded.

## 1D — Outbox Queue

Every offline write (open session, add consumption, close session) is added to the outbox:
  {
    action: 'open_session' | 'add_consumption' | 'close_session',
    payload: ,
    status: 'pending',
    retries: 0,
    created_at: ISO timestamp
  }

Processing rules:
  - Process in FIFO order (by created_at ascending)
  - On success: delete the outbox entry, update the local record as synced: true
  - On failure: increment retries. After 3 failures: set status = 'failed', show an alert to admin
  - Never process a later action for a session if an earlier action for that session is still pending (dependency ordering)
  - Wrap the entire sync run in a try/catch — one failed item must not block the rest

## 1E — Conflict Resolution

When syncing a session that was opened offline:
  - Before inserting, check if client_id already has an open session in Supabase
  - If yes: do NOT auto-resolve. Create a conflict entry and show it to the admin in a "Conflits à résoudre" screen
  - Admin can choose: keep local session, discard local session, or merge (keep both as separate sessions)
  - After resolution: remove conflict entry and continue sync

## 1F — What Works Offline

These actions must work fully offline (no degraded mode):
  - Open a new session (time or consumption mode)
  - View all currently open sessions (from IndexedDB cache)
  - Add consumptions to a session
  - Close / checkout a session (calculate locally, queue sync)
  - View the product list (from cache)

These actions require connectivity and must show a clear "Non disponible hors ligne" message:
  - Create a new client account
  - View historical transactions and reports
  - Change venue settings
  - Staff management

## 1G — Data Freshness

When the app comes online:
  1. Run the outbox sync (upload pending writes)
  2. Refresh local caches: pull latest products, clients, open sessions from Supabase
  3. Update lastSyncAt timestamp
  4. Show a toast: "Synchronisation réussie — X éléments mis à jour"

TASK 2 — CRASH-RESILIENT BILLING ENGINE

## 2A — Core Principle: Server Start Time, Always

The billing engine MUST NOT calculate duration as (current_time - in_memory_start_time). That approach fails on app crash, phone shutdown, browser tab close, or refresh.

The correct approach:
  - When a session is opened, write start_time to BOTH IndexedDB AND Supabase immediately
  - start_time is always a server-generated ISO timestamp (use Supabase's NOW() via RPC or insert with server_now())
  - Never use new Date() on the client as the authoritative start_time — only use it as a fallback if the server write has not yet confirmed (offline case)
  - If offline: use new Date().toISOString() as a provisional start_time and store it locally. Mark the session as provisional_start: true. When synced, replace with the server timestamp.
  - Every time the app loads or the session list is rendered, compute duration from: now - session.start_time (pulled from DB/local cache), not from any in-memory counter

## 2B — Timer Display

The UI timer is a display-only counter. It does NOT drive billing calculations:
  - On mount: read start_time from the session object (from DB/cache)
  - Display: elapsed = Math.floor((Date.now() - new Date(start_time).getTime()) / 1000)
  - Tick every second with setInterval
  - On unmount: clear the interval
  - If the component remounts (navigation, crash recovery, refresh): the timer picks back up from the real start_time — no drift, no reset

This means a session that started 2 hours ago will show 2:00:xx correctly even if the app was closed for 1:55 of those 2 hours.

## 2C — Duration & Amount Calculation

At checkout, calculate the final amount using ONLY persisted timestamps:
  - Pull start_time and end_time from the session record (DB or local cache)
  - duration_seconds = Math.max(0, Math.floor((new Date(end_time) - new Date(start_time)) / 1000))
  - duration_minutes = duration_seconds / 60
  - billed_units = duration_minutes / rate_unit_minutes    // e.g. rate_unit = 30 → per 30 min
  - amount = Math.max(0, Math.round(billed_units * rate * 100) / 100)   // rounded to 2 decimals, never negative

Flag any existing calculation that uses client-side Date() for start_time as: // BILLING BUG — replaced with persisted timestamp

## 2D — Edge Cases (all must be handled)

  - Session under 1 minute: bill minimum 1 unit (configurable per venue)
  - Session over 8 hours: bill normally, but flag in the breakdown as "session longue — vérifier"
  - Session crossing midnight: duration calculation is still correct because it uses ISO timestamps
  - App crash mid-session: on reopen, session is still in local DB with its original start_time — timer resumes correctly
  - Phone shutdown mid-session: same as above — start_time persisted in IndexedDB survives shutdown
  - Paused sessions: if pause is supported, store pause_start and pause_end timestamps in the session record. Deduct total paused duration from billing: effective_duration = total_duration - paused_duration. Never calculate pause gaps from memory.
  - Negative duration (clock skew): clamp to 0, log a warning // BILLING BUG if detected

## 3C — Consumption Billing Mode Behavior

During session:
  - Show live timer with label "Temps (non facturé)"
  - Staff adds products via a product picker bottom sheet
  - Running total = sum of all added products (this IS the bill)
  - Show itemized list of added products in real time

At checkout:
  - Staff reviews the itemized consumption list
  - Can edit quantities or remove items before finalizing
  - Final amount = sum of consumptions ONLY
  - Time cost is NEVER calculated, shown, or stored as a billable amount
  - balance_transactions.amount = consumption total
  - balance_transactions.breakdown = { billing_mode: 'consumption', items: [...], duration_minutes (informational only) }
